# Virtual Camera Router — спецификация проекта

## 1. Цель проекта

Разработать desktop-приложение для Windows, которое работает как универсальная прослойка между источниками видео и приложениями видеосвязи.

Приложение должно создавать виртуальную веб-камеру, видимую в Zoom, Discord, Telegram, браузерах и других приложениях как обычная камера.

Пользователь должен иметь возможность выбрать, какой видеопоток отдавать в виртуальную камеру:

- физическую веб-камеру;
- статичную картинку;
- видеофайл;
- элемент из локальной библиотеки медиа;
- fallback-заглушку, если источник недоступен.

---

## 2. Главный пользовательский сценарий

1. Пользователь запускает приложение.
2. Приложение регистрирует виртуальную камеру с именем `Chinaski Virtual Camera`.
3. Пользователь открывает Zoom.
4. В настройках Zoom выбирает `Chinaski Virtual Camera` как камеру.
5. В приложении пользователь выбирает источник:
   - физическая вебка;
   - картинка;
   - видео;
   - элемент из библиотеки.
6. Выбранный источник транслируется в Zoom как обычный видеопоток.

---

## 3. Важное уточнение по терминологии

Физические веб-камеры являются входными источниками.

Виртуальная камера приложения является выходным устройством.

Приложение не должно пытаться “выводить видео в физическую вебку”. Физическая вебка только читается.

Режим “использовать само приложение как вебку” означает, что приложение отдаёт выбранный пользователем контент в виртуальную камеру.

---

## 4. Целевая платформа MVP

MVP разрабатывается только под **Windows 11**.

Причина: основной нативный путь для виртуальной камеры — Windows Media Foundation Virtual Camera API.

Поддержка Windows 10, macOS и Linux не входит в MVP.

---

## 5. Технологический стек MVP

### 5.1 UI

- Tauri v2
- React
- TypeScript
- Tailwind CSS
- shadcn/ui или аналогичная компонентная библиотека

### 5.2 Backend приложения

- Rust backend внутри Tauri
- SQLite для хранения библиотеки и настроек
- SQLite FTS5 для поиска по библиотеке
- локальная папка приложения для хранения файлов, настроек и thumbnail-кэша

### 5.3 Виртуальная камера

- Windows Media Foundation Virtual Camera API
- C++ COM DLL с Custom Media Source
- регистрация виртуальной камеры через `MFCreateVirtualCamera`
- имя устройства: `Chinaski Virtual Camera`

### 5.4 Видео pipeline

Допустимые варианты:

- GStreamer — предпочтительно для MVP;
- FFmpeg/libav — для декодирования видео, извлечения кадров и генерации превью.

Видео pipeline должен уметь приводить любой источник к единому формату:

- 1280x720;
- 30 FPS;
- RGB32 или NV12;
- постоянный frame pacing;
- fallback frame при ошибке источника.

Разделы UI:

1. Dashboard
2. Sources
3. Library
4. Output
5. Settings
6. Diagnostics

---

## 6.2 Source Manager

Модуль управления источниками.

Типы источников:

- `PhysicalCameraSource`
- `ImageSource`
- `VideoFileSource`
- `LibraryItemSource`
- `FallbackSource`

Обязанности:

- получить список доступных физических камер;
- выбрать активный источник;
- переключать источник без перезапуска приложения;
- сообщать UI о текущем статусе источника;
- отдавать кадры в общий pipeline.

---

## 6.3 Virtual Camera Output

Модуль виртуальной камеры.

Обязанности:

- зарегистрировать виртуальную камеру в системе;
- запустить виртуальную камеру;
- остановить виртуальную камеру;
- удалить виртуальную камеру при необходимости;
- принимать кадры из pipeline;
- отдавать кадры в Media Foundation Custom Media Source.

Критерии:

- камера должна отображаться в Zoom как отдельное устройство;
- камера должна отображаться в системных приложениях, которые умеют выбирать webcam device;
- при отсутствии активного источника должна показываться fallback-заглушка.

---

## 6.4 Media Pipeline

Обязанности:

- декодировать видеофайлы;
- читать кадры с физической камеры;
- читать статичные изображения;
- приводить всё к единому разрешению/FPS;
- зацикливать видео при включённой опции loop;
- показывать последний валидный кадр при кратковременном сбое;
- отдавать fallback при критической ошибке.

MVP-формат выхода:

- 1280x720;
- 30 FPS;
- без звука;
- видео только.

---

## 6.5 Media Library

Локальная библиотека медиа.

Функции:

- добавить картинку;
- добавить видео;
- удалить элемент;
- переименовать элемент;
- добавить теги;
- поиск по названию;
- поиск по тегам;
- фильтр по типу: image/video;
- preview thumbnail;
- быстро выбрать элемент как текущий источник.

Хранимые поля:

- id;
- title;
- type;
- file_path;
- thumbnail_path;
- tags;
- duration;
- resolution;
- created_at;
- updated_at;
- last_used_at.

---

## 6.6 Diagnostics

Раздел диагностики.

Показывать:

- активный источник;
- активный FPS;
- задержку pipeline;
- статус виртуальной камеры;
- видит ли приложение физические камеры;
- ошибки Media Foundation;
- ошибки GStreamer/FFmpeg;
- лог последних событий.

Визуальный стиль:

- dark mode first;

- карточки источников;
- крупный preview;

- минималистичный dashboard;

Главный экран:

Статусы:

- `Virtual camera active`
- `Virtual camera stopped`
- `Source missing`
- `Streaming image`
- `Streaming video`
- `Streaming physical camera`
- `Fallback active`

## 9. Сценарии использования

## 9.1 Сценарий 1: Отдать картинку вместо вебки

1. Пользователь открывает приложение.
2. Добавляет картинку в библиотеку.
3. Выбирает картинку как источник.
4. Запускает виртуальную камеру.
5. Открывает Zoom.
6. Выбирает `Chinaski Virtual Camera`.
7. Zoom показывает выбранную картинку как вебку.

---

## 9.2 Сценарий 2: Отдать видео вместо вебки

1. Пользователь добавляет видеофайл.
2. Выбирает видео как источник.
3. Включает loop.
4. Запускает виртуальную камеру.
5. Zoom получает зацикленный видеопоток.

---

## 9.3 Сценарий 3: Прокинуть физическую вебку

1. Пользователь выбирает физическую камеру.
2. Приложение читает поток с камеры.
3. Приложение отдаёт этот поток в виртуальную камеру.
4. Zoom видит только виртуальную камеру.

---

## 9.4 Сценарий 4: Источник сломался

1. Активный файл удалён или камера отключена.
2. Приложение не падает.
3. Виртуальная камера продолжает работать.
4. В Zoom отображается fallback-заглушка.
5. UI показывает ошибку источника.

---

## 10. Архитектура

Общая схема:

```text
Physical Camera / Image / Video / Library Item
        ↓
Source Manager
        ↓
Media Pipeline
        ↓
Frame Normalizer
        ↓
Virtual Camera Frame Buffer
        ↓
Media Foundation Custom Media Source
        ↓
Zoom / Discord / Browser
```

UI не должен напрямую работать с Media Foundation.

UI общается с backend через Tauri commands.

Backend управляет состоянием приложения.

C++ COM-модуль отвечает только за системную виртуальную камеру и выдачу кадров наружу.

## 14. Технические риски

## 14.1 Риск 1: Виртуальная камера не видна в Zoom

Проверить:

- регистрация Media Foundation Virtual Camera;
- права доступа к камере в Windows Privacy Settings;
- корректное имя устройства;
- корректные категории устройства;
- запуск от администратора при необходимости.

---

## 14.2 Риск 2: Проблемы с Windows 10

Windows 10 не входит в MVP.

Не пытаться поддерживать Windows 10 в первой версии.

---

## 14.3 Риск 3: Камера блокируется другим приложением

Добавить диагностику:

- камера занята;
- нет доступа;
- ошибка открытия устройства.

---

## 14.4 Риск 4: Рывки FPS

Pipeline должен иметь frame scheduler.

Если источник отдаёт кадры нестабильно, приложение должно выдавать последний валидный кадр или fallback.

---

## 14.5 Риск 5: Большая задержка

Для MVP допустимая задержка: до 300 мс.

Цель: меньше 150 мс для физической камеры.

---

## 18. Инструкция агенту

Перед началом разработки агент должен:

1. Создать структуру проекта.
2. Подготовить Tauri + React + TypeScript frontend.
3. Подготовить Rust workspace для backend-модулей.
4. Подготовить C++/CMake проект для Media Foundation Virtual Camera.
5. Первым делом сделать research spike виртуальной камеры.
6. Не начинать UI-polish до подтверждения, что виртуальная камера видна в Zoom.

Главный приоритет разработки:

```text
Сначала доказать, что Zoom видит виртуальную камеру.
Потом научиться отдавать тестовый кадр.
Потом подключать источники.
Потом делать библиотеку.
Потом наводить красоту.
```
